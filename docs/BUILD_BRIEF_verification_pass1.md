# Build Brief — Verification Pipeline, Pass 1 (clean parse + parse-QA), ADR-013

**For:** Claude Code. **Governs:** `docs/DECISION_LOG.md` ADR-013 (read it first — §7 defects and §8
pipeline rules are binding), `FORMULATOR_SPECIFICATION.md`. **Nature:** this is a **data-engineering
task producing an auditable artifact**, not a UI change. Touch NO existing runtime code
(`blend-builder.js`, `herbuno-matrix.js` unchanged).

**Goal of Pass 1:** turn ONE supplier catalogue (thewholesaler.eu) into a clean, strictly-keyed
`botanical → commercial-forms` candidate dataset, with a parse-QA report — and run it on a
50-botanical proof cohort to **measure the real error rate and manual-review burden.** That
measurement is the go/no-go gate for the whole programme.

## 0. Non-negotiable rules (from ADR-013 §8)
- **No silent auto-merge.** Any ambiguous case → written to a review-queue file, never silently
  joined. Silent merging is what dropped Ashwagandha; it is prohibited.
- **Key strictly by normalised Latin binomial.** Common names are labels only, never merge keys.
- **Provenance:** every form observation records `source_supplier` + `observation_date`.
- Output is **UNVERIFIED candidate data** — label it as such; it is not production-ready.

## 1. Input
`allProductsthewholesaler_eu…csv` (the thewholesaler.eu export already in the project, ~29k rows).
Product header rows carry `Title` + `Type`; variant rows have blanks — process only rows with a
non-empty `Type` in the normalisation map.

## 2. Normalisation map (supplier `Type` → format code) — from ADR-013 §6
```
Extract Powder            → RE
Herbal/Fruit/Vegetable/Spice Powder → MP
Water Soluble Extract     → WL
Oil Soluble Extract       → OE
Propylene Glycol Extract  → WL-glycol
Glycerin Extract          → WL-glycerin
Extract Paste             → RE-paste
Oleoresin                 → OLR
Hydrosol                  → HYD
Infused Oil / Carrier Oil → CO
Herb,Cut / Tea & Infusions / Dried Flowers Cut → TC
Whole Herb/Spice, Seeds, Root, Dried Fruits/Flowers Whole → WD
```
Any `Type` not in this map → log to an `unmapped_types` report (do not guess).

## 3. The §7 parser fixes (all mandatory)
Title format observed: `[CC] Common Name <formwords> - Latin binomial(optional assay)`. e.g.
`[ID] Bacopa Extract (≥20% bacosides) - Bacopa monnieri`,
`Ashwagandha Glycerin Extract - Withania Somnifera`,
`[VN] Brahmi – Water hyssop Extract - Bacopa monnieri(≥20% Bacosides)`.

1. **Latin capitalisation** — normalise every binomial to `Genus species` (capital genus, lowercase
   species). "Withania Somnifera" and "Withania somnifera" MUST resolve to the same key.
2. **Strip assay from Latin** — remove trailing `(…)` / `≥x%` stuck to the binomial before keying.
3. **Multi-dash titles** — a title may contain several `-`/`–`. Extract the Latin from the LAST
   segment that matches a binomial pattern `^[A-Z][a-z]+ [a-z]+`, not the first dash.
4. **No fuzzy common-name merge** — do NOT join records by common-name substring/similarity. Only the
   normalised Latin key joins records.
5. **Missing Latin** — if no binomial is parseable, the record does NOT get a Latin key; route it to a
   `no_latin` review queue (do not force-merge it by common name).

## 4. Outputs (all written to a new `/knowledge/pass1/` folder, all committed)
- `botanical_candidates.json` — `{ "<Latin>": { common_names:[...], forms:[{code, source_supplier,
  observation_date}] } }`. Strictly Latin-keyed.
- `review_queue.json` — every ambiguity: case-collision candidates, assay-in-Latin cases, multi-dash
  uncertainty, no-Latin records, near-duplicate Latin keys (e.g. genus match, species differs by 1
  char). Each entry: the raw title(s), the reason flagged, the candidate interpretation(s).
- `parse_qa_report.md` — counts: rows processed, records mapped, distinct Latin keys, unmapped types,
  no-Latin count, review-queue size; and the **spot-check block below**.
- `unmapped_types.json` — any `Type` value not in the map, with counts.

## 5. Mandatory spot-check (must appear in parse_qa_report.md)
These MUST resolve correctly or Pass 1 has failed:
- `Withania somnifera` (Ashwagandha) — present, forms aggregated (was dropped by the buggy parser).
- `Bacopa monnieri` (Bacopa/Brahmi) — single key, all forms aggregated (was scattered across 3 keys).
- `Asparagus racemosus` (Shatavari) — present with its full form set.
- `Centella asiatica` (Gotu Kola) — present.
Print each with its aggregated forms so a human can eyeball correctness.

## 6. The 50-botanical proof cohort + cost measurement (the actual point)
- Select the 50 botanicals with the most product listings (highest commercial signal).
- For those 50, produce the clean records AND manually-reviewable output.
- **Record in `parse_qa_report.md`:**
  - parse error rate on the 50 (records needing manual correction ÷ 50),
  - review-queue entries generated by the 50 (the manual-review burden),
  - an honest estimate of minutes of human review the 50 would need to reach "identity-verified."
- This number is the **go/no-go gate** for scaling to the full ~700. Do not proceed to Pass 2 or
  more suppliers until the owner has seen it.

## 7. What NOT to do
- Do not call any external API (Pass 2 authority cross-check is a LATER pass).
- Do not layer other suppliers yet (Pass 3).
- Do not modify `herbuno-matrix.js`, `blend-builder.js`, or any runtime code.
- Do not mark anything "verified" — Pass 1 output is candidate/UNVERIFIED by definition.
